The single-agent problem

One pass, one perspective

  • Converges too early
  • Sticks with the first framing
  • Stops digging at a plausible answer
  • Struggles to review its own work
  • Sounds confident even when wrong
  • Won’t push back on a bad premise

Single agents aren’t broken. Some tasks just need more than one perspective, role, or pass.

The multi-agent reframe
Multi-agent is NOT
“More agents = better.” Bolting on copies adds cost, not quality.
→
Multi-agent IS
Different agents, structured to improve the outcome. Every role moves the result forward.
specializationindependent reviewadversarial critiqueparallel explorationorchestrationtool specializationQAstructured handoffs

The second agent has to earn its place.

A coordinated workflow of model-driven actors toward a shared goal

Failure modes

When coordination costs more than the task

  • Runaway token cost
  • Infinite loops
  • Agents just agreeing with each other
  • Debate becoming rhetoric, not truth-seeking
  • Lost context across handoffs
  • Unclear ownership · no stopping condition
  • Too many agents for a simple job
  • Debugging five agents at 11 p.m.

If you can’t explain how the next agent improves the result — don’t add it.

The decision
Use multi-agent when…
  • Multiple perspectives or independent review help
  • Adversarial critique or role specialization matters
  • Parallel exploration, complex handoffs, or QA
  • Long-horizon execution
Skip it when…
  • One prompt already works
  • Speed matters more than quality
  • The task is simple or easy to verify by hand
  • The agents don’t have distinct jobs
